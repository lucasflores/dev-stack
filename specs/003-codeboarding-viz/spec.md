# Feature Specification: CodeBoarding Visualization

**Feature Branch**: `003-codeboarding-viz`  
**Created**: 2026-03-04  
**Status**: Draft  
**Input**: User description: "Replace dev-stack's D2-based visualization module with a CodeBoarding CLI wrapper that generates Mermaid.js architecture diagrams, and add per-folder README sub-diagrams as a new capability."

## Clarifications

### Session 2026-03-04

- Q: How should dev-stack discover Mermaid diagrams in CodeBoarding's output directory? → A: Parse the authoritative index file `.codeboarding/analysis.json` that maps components to their Mermaid diagram file paths, decoupling dev-stack from CodeBoarding's internal directory layout.
- Q: How should dev-stack handle CodeBoarding failures due to missing/invalid API keys? → A: All CodeBoarding non-zero exits are treated uniformly — capture stderr and display it verbatim to the user. No special-case detection of API key or authentication errors; CodeBoarding's own error messages guide the user.
- Q: How should uninstall discover all previously-injected README managed sections? → A: Maintain a ledger file (e.g., `.codeboarding/injected-readmes.json`) that records every README path written to during visualization. On uninstall, iterate the ledger and remove managed sections from each listed file.
- Q: What subprocess timeout should dev-stack apply when invoking CodeBoarding CLI? → A: 300 seconds (5 minutes) by default, configurable via a `--timeout` CLI flag.
- Q: What should the default `--depth-level` be when the flag is omitted? → A: Default to `--depth-level 2`, producing the top-level architecture diagram plus one level of component sub-diagrams. CodeBoarding's own default is depth 1 (top-level only), which would not satisfy the per-folder sub-diagram requirement. Sub-diagrams are generated by default at depth 2.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Run `dev-stack visualize` to Generate Architecture Diagrams (Priority: P1)

A developer navigates to their project root, runs `dev-stack visualize`, and the system invokes the CodeBoarding CLI as a subprocess. CodeBoarding performs LSP-based static analysis and LLM-driven architecture decomposition on the repository. Once complete, the top-level Mermaid architecture diagram is extracted from CodeBoarding's `.codeboarding/` output and injected as an inline Mermaid code block into the repository's root `README.md` using managed section markers. The developer sees the diagram rendered natively on GitHub or GitLab with no additional tooling.

**Why this priority**: This is the primary workflow — generating and surfacing a top-level architecture diagram is the minimum viable replacement for the existing D2 visualization pipeline. Without this, the feature delivers no value.

**Independent Test**: Run `dev-stack visualize` in a repository with Python source files, confirm that CodeBoarding is invoked, `.codeboarding/` output is produced, and a Mermaid code block appears inside managed markers in the root `README.md`.

**Acceptance Scenarios**:

1. **Given** a repository with source files and CodeBoarding CLI installed, **When** the developer runs `dev-stack visualize`, **Then** CodeBoarding is invoked as a subprocess, its `.codeboarding/` output directory contains analysis results, and a Mermaid architecture diagram code block is injected into the root `README.md` between `=== DEV-STACK:BEGIN:architecture ===` and `=== DEV-STACK:END:architecture ===` markers (wrapped in HTML comments for `.md` files).
2. **Given** the root `README.md` already contains user-authored content, **When** `dev-stack visualize` runs, **Then** existing content outside the managed section markers is untouched and the Mermaid block is inserted or replaced idempotently within the markers.
3. **Given** no root `README.md` exists, **When** `dev-stack visualize` runs, **Then** a new `README.md` is created containing only the managed Mermaid diagram section.
4. **Given** CodeBoarding CLI is not installed, **When** the developer runs `dev-stack visualize`, **Then** the command exits with a clear error message stating that CodeBoarding CLI is required and provides installation guidance.

---

### User Story 2 — Per-Folder Sub-Diagrams (Priority: P2)

After running `dev-stack visualize`, each folder that CodeBoarding identifies as a distinct component receives its own `README.md` with an inline Mermaid sub-diagram. This provides developers with drill-down documentation directly in the codebase, showing the internal architecture of each component folder.

**Why this priority**: Sub-diagrams add significant navigability and documentation depth, but they build on top of the core CodeBoarding invocation delivered in P1. They are a differentiating capability that the old D2 pipeline did not offer.

**Independent Test**: Run `dev-stack visualize` on a multi-package repository, then verify that at least one component folder has a `README.md` containing a Mermaid sub-diagram within managed section markers.

**Acceptance Scenarios**:

1. **Given** a repository with multiple component folders identified by CodeBoarding, **When** the developer runs `dev-stack visualize`, **Then** each identified component folder contains a `README.md` with a Mermaid sub-diagram inside managed markers (`=== DEV-STACK:BEGIN:component-architecture ===` / `=== DEV-STACK:END:component-architecture ===`, wrapped in HTML comments).
2. **Given** a component folder already has a user-authored `README.md`, **When** sub-diagrams are injected, **Then** the existing content outside managed markers is preserved.
3. **Given** a component folder has no existing `README.md`, **When** sub-diagrams are injected, **Then** a new `README.md` is created with the sub-diagram as its content.

---

### User Story 3 — Incremental Mode (Priority: P3)

A developer makes changes to a few files and runs `dev-stack visualize --incremental`. The system detects which files changed via the existing hash-based manifest, passes `--incremental` to CodeBoarding, and only re-analyzes the affected components. This saves time and LLM API costs on large repositories.

**Why this priority**: Incremental analysis is an optimization that adds efficiency but is not required for core functionality. Full re-analysis works correctly without it.

**Independent Test**: Modify a single file after an initial visualization run, then run `dev-stack visualize --incremental` and verify that only the changed component's diagram is updated and the CodeBoarding CLI receives the `--incremental` flag.

**Acceptance Scenarios**:

1. **Given** a prior visualization run has completed and the manifest is stored, **When** a developer changes one source file and runs `dev-stack visualize --incremental`, **Then** the system detects the changed file via the manifest, passes `--incremental` to CodeBoarding, and only the affected component diagrams are updated.
2. **Given** no previous manifest exists, **When** the developer runs `dev-stack visualize --incremental`, **Then** the system performs a full analysis (equivalent to running without `--incremental`).
3. **Given** no files have changed since the last run, **When** the developer runs `dev-stack visualize --incremental`, **Then** the system reports that all diagrams are up to date and does not invoke CodeBoarding.

---

### User Story 4 — Depth Level Control (Priority: P3)

A developer runs `dev-stack visualize --depth-level 3` to control how many levels of component decomposition CodeBoarding produces. A depth of 1 produces only the top-level architecture diagram; higher values produce increasingly detailed sub-component breakdowns. When `--depth-level` is omitted, dev-stack defaults to `--depth-level 2` (top-level plus one level of sub-components), which satisfies both the root diagram and per-folder sub-diagram requirements.

**Why this priority**: Depth control is a configuration refinement that lets teams tune verbosity. The default behavior (depth 2) works without it.

**Independent Test**: Run `dev-stack visualize --depth-level 1` and confirm only the top-level diagram is generated; run with `--depth-level 3` and confirm deeper sub-diagrams appear.

**Acceptance Scenarios**:

1. **Given** a repository with deeply nested components, **When** the developer runs `dev-stack visualize --depth-level 1`, **Then** only the top-level architecture diagram is generated and no per-folder sub-diagrams are created.
2. **Given** a repository with deeply nested components, **When** the developer runs `dev-stack visualize --depth-level 3`, **Then** sub-diagrams are generated down to three levels of component decomposition.

---

### User Story 5 — Analysis-Only Mode (Priority: P3)

A developer runs `dev-stack visualize --no-readme` to run CodeBoarding analysis without injecting diagrams into any README files. The `.codeboarding/` output is produced for manual inspection or external tooling consumption.

**Why this priority**: This is a convenience flag for advanced users who want the raw analysis without README modification. It can be tested independently but adds limited value for most workflows.

**Independent Test**: Run `dev-stack visualize --no-readme`, verify `.codeboarding/` output exists, and confirm no README files were created or modified.

**Acceptance Scenarios**:

1. **Given** a repository with source files, **When** the developer runs `dev-stack visualize --no-readme`, **Then** CodeBoarding analysis runs and `.codeboarding/` output is produced, but no README files are created or modified.

---

### User Story 6 — Module Lifecycle (Priority: P1)

The VisualizationModule maintains its `ModuleBase` lifecycle contract (`install`, `uninstall`, `update`, `verify`). Installation ensures the `.codeboarding/` output directory exists and verifies that the CodeBoarding CLI binary is available on PATH. Uninstallation cleans up managed directories and removes managed README sections. Verification reports the health status including CodeBoarding CLI availability.

**Why this priority**: The module lifecycle is foundational to dev-stack's architecture. Without it, visualization cannot be installed, verified, or cleanly removed.

**Independent Test**: Run `dev-stack init` with visualization enabled, verify CodeBoarding CLI is detected, then run `dev-stack status` to confirm healthy status.

**Acceptance Scenarios**:

1. **Given** the CodeBoarding CLI is installed on PATH, **When** the developer runs `dev-stack init` with visualization enabled, **Then** the module installs successfully, creates `.codeboarding/` state directory, and reports success.
2. **Given** the CodeBoarding CLI is not installed, **When** the module's `verify()` is called, **Then** it reports unhealthy status with a clear message indicating CodeBoarding CLI is required and provides installation guidance.
3. **Given** the visualization module is installed, **When** the developer uninstalls it, **Then** the `.codeboarding/` directory and all managed README sections are cleaned up.
4. **Given** the visualization module is already installed, **When** the developer runs update, **Then** the module re-verifies CodeBoarding CLI availability and refreshes its configuration.

### Edge Cases

- What happens when CodeBoarding CLI exits with a non-zero exit code (including API key errors, analysis failures, etc.)? The system captures stderr, displays it verbatim to the user, and exits gracefully without modifying any README files. No special-case detection is performed — all failure modes are treated uniformly.
- What happens when CodeBoarding output is malformed or missing expected Mermaid content? The system logs a warning, skips README injection for the affected component, and reports partial success.
- What happens when the repository has no analyzable source files? CodeBoarding may produce empty output; the system reports "no components found" and does not inject empty diagram blocks.
- What happens when a managed README section from a previous version (D2-based) exists? The system replaces it with the new Mermaid content within the same managed markers, ensuring a clean migration.
- What happens when the `--depth-level` value exceeds the actual nesting depth of the repository? The system treats the excess depth as a no-op and generates diagrams only to the depth that components exist.
- What happens when disk permissions prevent writing to a folder's README? The system logs a warning for that folder and continues processing remaining folders.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST invoke CodeBoarding CLI as a subprocess (`codeboarding --local <repo-path>`) and never import it as a Python library.
- **FR-002**: System MUST parse the index file `.codeboarding/analysis.json` produced by CodeBoarding that maps each component to its Mermaid diagram file path, and extract diagram content from the referenced component markdown files.
- **FR-003**: System MUST inject the top-level architecture Mermaid diagram as an inline code block into the repository root `README.md` using the brownfield managed-section markers (`DEV-STACK:BEGIN:architecture` / `DEV-STACK:END:architecture`).
- **FR-004**: System MUST inject per-folder component sub-diagrams as inline Mermaid code blocks into each identified component folder's `README.md` using managed-section markers (`DEV-STACK:BEGIN:component-architecture` / `DEV-STACK:END:component-architecture`).
- **FR-005**: System MUST create `README.md` files in component folders that do not already have one, when sub-diagrams are generated for those folders.
- **FR-006**: System MUST support an `--incremental` flag that uses the existing hash-based manifest to detect changed files and passes `--incremental` to CodeBoarding CLI.
- **FR-007**: System MUST support a `--depth-level <N>` flag that controls the number of decomposition levels passed to CodeBoarding CLI. When omitted, dev-stack MUST default to `--depth-level 2` (top-level architecture plus one level of component sub-diagrams). CodeBoarding's own default is depth 1 (top-level only), which is insufficient for per-folder sub-diagram generation.
- **FR-008**: System MUST support a `--no-readme` flag that skips all README injection while still producing `.codeboarding/` analysis output.
- **FR-009**: System MUST exit with a clear, actionable error message when the CodeBoarding CLI binary is not found on PATH.
- **FR-010**: System MUST completely remove the D2 CLI dependency — no D2 installation, version checking, or rendering logic remains.
- **FR-011**: The `VisualizationModule` MUST implement the full `ModuleBase` lifecycle contract: `install()`, `uninstall()`, `update()`, and `verify()`.
- **FR-012**: `install()` MUST create the `.codeboarding/` state directory and verify that the CodeBoarding CLI is available on PATH.
- **FR-013**: `uninstall()` MUST remove the `.codeboarding/` state directory, the `docs/diagrams/` directory (legacy), and all managed README sections listed in the injection ledger file (`.codeboarding/injected-readmes.json`).
- **FR-014**: `verify()` MUST report whether the CodeBoarding CLI binary is present and return healthy/unhealthy status accordingly.
- **FR-015**: README injection MUST be idempotent — running visualization multiple times produces the same result as running it once (no duplicated sections, no content drift).
- **FR-016**: System MUST preserve all existing user-authored content in README files outside of managed section markers.
- **FR-017**: System MUST produce structured JSON output when `--json` is passed, consistent with dev-stack's CLI contract.
- **FR-018**: System MUST handle all CodeBoarding CLI failures (non-zero exit, timeout) uniformly — capture stderr, display it verbatim to the user, and exit without modifying any README files. No special-case parsing of error categories (e.g., API key errors) is performed; CodeBoarding's own messages guide the user.
- **FR-019**: The system MUST maintain an injection ledger file (`.codeboarding/injected-readmes.json`) that records the path of every README file into which a managed section was written. The ledger MUST be updated on each visualization run to reflect the current set of injected files.
- **FR-020**: System MUST apply a subprocess timeout of 300 seconds (5 minutes) by default when invoking CodeBoarding CLI. A `--timeout <seconds>` flag MUST allow the user to override this default.

### Key Entities

- **CodeBoarding Output**: The `.codeboarding/` directory produced by the CodeBoarding CLI, containing analysis results and Mermaid diagram code for the repository's architecture at various levels of decomposition. Includes the authoritative index file `analysis.json` that maps each component to its diagram file path.
- **Managed Section**: A region in a README file delimited by `DEV-STACK:BEGIN:<id>` and `DEV-STACK:END:<id>` markers, owned by dev-stack and safe to overwrite without affecting surrounding user content.
- **Architecture Diagram**: A top-level Mermaid diagram representing the repository's major components and their relationships.
- **Component Sub-Diagram**: A Mermaid diagram scoped to a specific folder/component, showing its internal structure and relationships.
- **Manifest**: The existing hash-based file change tracking mechanism used to detect which files have been modified since the last visualization run.
- **Injection Ledger**: A file (`.codeboarding/injected-readmes.json`) that records the paths of all README files into which dev-stack has injected managed Mermaid diagram sections. Used by uninstall to locate and clean up all managed sections without a full repository tree walk.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A developer can generate and view an architecture diagram in their repository's README within 5 minutes of running `dev-stack visualize` for the first time, with no tooling beyond dev-stack and CodeBoarding CLI.
- **SC-002**: Running `dev-stack visualize` twice in succession on an unchanged repository produces byte-identical README files (idempotency).
- **SC-003**: Per-folder sub-diagrams are generated for all CodeBoarding-identified components, providing drill-down documentation without manual authoring effort.
- **SC-004**: When no files have changed since the last run, incremental mode completes without invoking CodeBoarding (< 1s dev-stack overhead). When files have changed, the `--incremental` flag is correctly passed to CodeBoarding CLI.
- **SC-005**: Existing README content outside managed markers is never modified or lost across any combination of install, visualize, and uninstall operations.
- **SC-006**: The `dev-stack status` command accurately reports visualization module health, including CodeBoarding CLI availability, after install and after uninstall.
- **SC-007**: D2 CLI is fully removed as a dependency — no install attempts, no version checks, no D2-related code paths remain.
- **SC-008**: All CLI flags (`--incremental`, `--depth-level`, `--no-readme`, `--timeout`, `--json`, `--verbose`) are functional and produce the expected behavior as defined in the acceptance scenarios.

## Assumptions

- CodeBoarding CLI is installed separately by the user (e.g., via `pipx install codeboarding`) and is available on PATH. dev-stack does not manage CodeBoarding's installation or its Python environment.
- CodeBoarding CLI requires its own LLM API key (e.g., `GOOGLE_API_KEY` for Google Gemini free tier). dev-stack does not configure or validate this key — CodeBoarding handles its own LLM orchestration and will report its own errors if the key is missing.
- CodeBoarding produces the authoritative index file `.codeboarding/analysis.json` that maps each identified component to its Mermaid diagram file path within the `.codeboarding/` output directory. dev-stack parses this index rather than scanning the directory directly, decoupling it from CodeBoarding's internal layout.
- The `--incremental` and `--depth-level` flags map to CodeBoarding CLI flags or arguments. If CodeBoarding does not yet support these flags, the implementation will document the expected interface and degrade gracefully.
- GitHub and GitLab natively render Mermaid code blocks in Markdown, so no SVG/PNG rendering step is needed.
- The existing brownfield managed-section markers system is sufficient for README injection without modification.
- Legacy D2-related files (`.dev-stack/viz/overview.d2`, `docs/diagrams/`) from previous installations can be safely cleaned up during uninstall or migration.

## Out of Scope

- Direct LLM API integration in dev-stack (CodeBoarding handles its own LLM calls).
- Importing CodeBoarding as a Python library (subprocess invocation only).
- SVG/PNG rendering of Mermaid diagrams (GitHub and GitLab render Mermaid natively).
- Modifying the pre-commit pipeline stages.
- Changes to AgentBridge or commit message generation.
- Managing CodeBoarding's LLM API key configuration.
- Automatic installation of the CodeBoarding CLI tool itself.
