# Feature Specification: Init Onboarding Fixes

**Feature Branch**: `007-init-onboarding-fixes`  
**Created**: 2026-03-10  
**Status**: Draft  
**Input**: Resolve greenfield init UX friction, DEV_STACK_AGENT=none ignored, detect-secrets circular self-detection, README documentation gaps, constitution template misplacement, and brownfield mode mislabeling.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Security Stage Handles Dev-Stack Files Without Circular Detection (Priority: P1)

A new user runs `dev-stack init` and then attempts their first commit. The pre-commit hook triggers the pipeline, which includes the security stage. The security stage scans the repo using `detect-secrets` and encounters SHA-256 checksums in `.dev-stack/hooks-manifest.json`. The stage recognizes these as known dev-stack artifacts and excludes them from scanning. The `.secrets.baseline` file does not contain self-referential entries. The commit proceeds without the user needing `--no-verify`.

**Why this priority**: This is a BLOCKING issue. Without this fix, no new user can make a normal commit after initialization. The security stage is a hard gate, and its circular self-detection creates an infinite regression that cannot be resolved through auditing. Every other fix is moot if users cannot commit.

**Independent Test**: Run `dev-stack init` in a fresh repo, then run `dev-stack pipeline run`. Verify the security stage passes. Inspect `.secrets.baseline` and confirm it contains no entries from `.dev-stack/` paths and no self-referential `hashed_secret` entries.

**Acceptance Scenarios**:

1. **Given** a freshly initialized repo with `.dev-stack/hooks-manifest.json` containing SHA-256 checksums, **When** the security stage runs `detect-secrets scan`, **Then** files under `.dev-stack/` are excluded from scanning and the baseline contains no entries for those paths.
2. **Given** a `.secrets.baseline` file generated by init, **When** the security stage runs `detect-secrets scan --baseline .secrets.baseline`, **Then** the baseline file itself does not produce new self-referential findings.
3. **Given** a repo with a real secret (e.g., an API key in a source file), **When** the security stage runs, **Then** it correctly reports FAIL with unaudited findings for that file.

### User Story 2 — Greenfield Init After uv init --package Succeeds Without --force (Priority: P1)

A new user follows the README greenfield workflow: they run `uv init --package`, then run `dev-stack init`. The init command detects the `uv init --package` predecessor pattern, auto-resolves all files created by `uv init --package` as greenfield predecessors, and completes installation without requiring `--force`. The user sees a clean success result with no conflict warnings.

**Why this priority**: This is the primary onboarding path documented in the README. Requiring `--force` for the expected workflow is confusing and creates an immediate negative first impression. A seamless greenfield flow is essential for adoption.

**Independent Test**: Create a new directory, run `uv init --package myproject`, then run `dev-stack --json init`. Verify exit code 0 and that the JSON output shows no pending conflicts.

**Acceptance Scenarios**:

1. **Given** a directory initialized with `uv init --package`, **When** the user runs `dev-stack init` (without --force), **Then** init completes successfully with exit code 0.
2. **Given** a directory initialized with `uv init --package` plus additional user-created files that conflict, **When** the user runs `dev-stack init`, **Then** init reports only the non-predecessor files as conflicts and exits with code 3.
3. **Given** a directory initialized with `uv init --package`, **When** the user runs `dev-stack --json init`, **Then** the JSON output lists all `uv init` files with resolution `greenfield_predecessor`.

### User Story 3 — DEV_STACK_AGENT=none Skips Agent Detection (Priority: P2)

A user who does not use any AI coding agent sets `DEV_STACK_AGENT=none` before running `dev-stack init`. The init command honors this override and records `agent.cli = "none"` with no path in the manifest. No agent-dependent pipeline stages are configured or run.

**Why this priority**: The README documents this as a supported workflow. Ignoring an explicit user override violates the principle of least surprise and forces agent dependencies on users who opted out.

**Independent Test**: Run `DEV_STACK_AGENT=none dev-stack --json init --force` in an initialized repo. Verify the JSON output shows `"cli": "none"` and `"path": null` for the agent field.

**Acceptance Scenarios**:

1. **Given** `DEV_STACK_AGENT=none` is set, **When** the user runs `dev-stack init`, **Then** agent detection returns `cli="none"` and `path=None` regardless of what agents are installed on the system.
2. **Given** `DEV_STACK_AGENT=none` is set and `gh copilot` is available, **When** the user runs `dev-stack --json init`, **Then** the manifest records `agent.cli = "none"`, not `"copilot"`.

### User Story 4 — README Documents Initial Commit Workflow (Priority: P2)

A new user following the README reaches the "commit the generated files" step. The documentation shows a clean `git add && git commit` workflow — no `--no-verify` or `--force` flags required — because the security stage exclusions (US1) prevent false positives on dev-stack's own managed files. The user completes their first commit without confusion.

**Why this priority**: A documentation gap at the most critical step (first commit) causes users to get stuck and lose confidence in the tool. This directly impacts onboarding success rates.

**Independent Test**: Follow the README instructions verbatim in a fresh repo. Verify that every step succeeds without undocumented workarounds.

**Acceptance Scenarios**:

1. **Given** a user following the README greenfield instructions, **When** they reach the initial commit step, **Then** the documentation shows a standard `git add && git commit` workflow with no special flags required.
2. **Given** a freshly initialized repo where the security stage exclusions are in place, **When** the user runs `git commit` for the first time, **Then** the commit succeeds without `--no-verify`.

### User Story 5 — Baseline Practices Merge Into SpecKit Constitution (Priority: P2)

A user installs dev-stack with both the `vcs_hooks` and `speckit` modules. The baseline practices (atomic commits, branch naming, etc.) are injected as clauses into the speckit constitution template at `.specify/templates/constitution-template.md`. No standalone `constitution-template.md` file appears at the repo root.

**Why this priority**: A floating root-level file violates the design intent — baseline practices should integrate with the speckit workflow, not exist in parallel. This causes confusion about which file governs project practices.

**Independent Test**: Run `dev-stack init` in a new repo. Verify that no `constitution-template.md` exists at the repo root. Verify that `.specify/templates/constitution-template.md` contains the baseline practices content from the `vcs_hooks` module.

**Acceptance Scenarios**:

1. **Given** a fresh repo, **When** `dev-stack init` runs with both `vcs_hooks` and `speckit` modules, **Then** no `constitution-template.md` file exists at the repo root.
2. **Given** a fresh repo, **When** `dev-stack init` completes, **Then** `.specify/templates/constitution-template.md` contains the dev-stack baseline practices (atomic commits, branch naming conventions, etc.).
3. **Given** a repo that was previously initialized with a root-level `constitution-template.md`, **When** `dev-stack init` runs again (reinit), **Then** the root file is migrated into the speckit template and the root file is removed.

### User Story 6 — Greenfield Mode Correctly Labeled in Manifest (Priority: P3)

A user follows the greenfield flow (`uv init --package` → `dev-stack init`). The `dev-stack.toml` manifest records `mode = "greenfield"` because the `uv init --package` files are recognized as known predecessors and do not count as brownfield conflicts.

**Why this priority**: While cosmetic, incorrect mode labeling undermines user trust and could cause confusion when debugging pipeline behavior or reading manifest files.

**Independent Test**: Run the greenfield flow and inspect `dev-stack.toml`. Verify `mode = "greenfield"`.

**Acceptance Scenarios**:

1. **Given** a directory initialized with `uv init --package` and no other pre-existing files, **When** `dev-stack init` completes, **Then** `dev-stack.toml` records `mode = "greenfield"`.
2. **Given** a directory initialized with `uv init --package` plus additional user-modified files, **When** `dev-stack init` completes with `--force`, **Then** `dev-stack.toml` records `mode = "brownfield"`.

### Edge Cases

- What happens when `detect-secrets` is not installed? The security stage should skip gracefully (already handled — baseline is not generated if `detect-secrets` is not found).
- What happens when `uv init --package` output changes in a future `uv` version? The predecessor detection should fail gracefully and fall back to normal conflict reporting.
- What happens when `DEV_STACK_AGENT` is set to an invalid/unknown agent name? The system should report a clear error or fall through to auto-detection (current behavior is to fall through — this is acceptable).
- What happens when a user has a legitimate file named `constitution-template.md` at the repo root that is NOT from dev-stack? The migration logic should only act on files that match dev-stack's known content signature.
- What happens when `vcs_hooks` is installed without the `speckit` module? Baseline practices content is skipped entirely — no constitution file is created anywhere.
- What happens when the `.secrets.baseline` contains findings from user code alongside dev-stack false positives? Only the dev-stack paths should be excluded; user-code findings must still be evaluated.

## Requirements *(mandatory)*

### Functional Requirements

**High — Security Stage (Issue 3)**

- **FR-001**: The secrets baseline generation and the security stage scan MUST pass `--exclude-files` CLI arguments to `detect-secrets scan` to exclude the `.dev-stack/` directory so that SHA-256 checksums in `hooks-manifest.json` are never recorded as findings.
- **FR-002**: The secrets baseline generation and the security stage scan MUST pass `--exclude-files` CLI arguments to `detect-secrets scan` to exclude the `.secrets.baseline` file itself to prevent self-referential detection loops.
- **FR-003**: The security stage MUST continue to detect and report genuine secrets in user source files.

**High — Greenfield Init Flow (Issue 1)**

- **FR-004**: When all detected conflicts are resolved as `greenfield_predecessor`, the init command MUST proceed without requiring `--force`.
- **FR-005**: The `has_existing_conflicts` check MUST exclude files whose resolution is `greenfield_predecessor` when determining whether conflicts block initialization.

**Medium — Agent Detection Override (Issue 2)**

- **FR-006**: When `DEV_STACK_AGENT` is set to `"none"` (case-insensitive), agent detection MUST immediately return `AgentInfo(cli="none", path=None)` without attempting binary resolution or falling through to auto-detection.

**Medium — README Documentation (Issue 4)**

- **FR-007**: The README greenfield instructions MUST document the first commit as a clean step (no `--no-verify` required) since the security stage exclusions resolve the blocking issue. A troubleshooting section MUST mention `--no-verify` as a fallback for edge cases only.
- **FR-008**: The README MUST NOT document `--force` in the greenfield path since predecessor auto-resolution (FR-004/FR-005) eliminates the need for it.

**Medium — Constitution Template Placement (Issue 5)**

- **FR-009**: The `vcs_hooks` module MUST NOT create a standalone `constitution-template.md` at the repo root.
- **FR-010**: The `vcs_hooks` module MUST inject its baseline practices content into `.specify/templates/constitution-template.md` when the speckit module is also installed. When speckit is not installed, the baseline practices content MUST be skipped entirely (no file created).
- **FR-011**: On reinit, if a root-level `constitution-template.md` exists from a prior installation, the system MUST migrate its content into the speckit template and remove the root file.

**Low — Mode Label (Issue 6)**

- **FR-012**: The mode determination logic MUST classify the init as `"greenfield"` when all conflicts are resolved as `greenfield_predecessor` and the repo was not previously initialized.

### Key Entities

- **Conflict Report**: A collection of file conflicts detected during init; each conflict has a path, current hash, expected hash, and resolution status (`pending`, `greenfield_predecessor`, `accepted`, `skipped`, `merged`, or `overwritten`).
- **Secrets Baseline**: A JSON file (`.secrets.baseline`) that records known findings from `detect-secrets scan`; each finding can be audited as `is_secret: true` or `is_secret: false`.
- **Agent Info**: The detected coding agent configuration; has a CLI identifier and optional filesystem path.
- **Constitution Template**: A markdown file (`.specify/templates/constitution-template.md`) containing project governance practices; serves as the scaffold for new constitutions and receives injected clauses from other modules.
- **Init Mode**: Classification of the target repository as `greenfield`, `brownfield`, or `reinit`.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A new user completes the full greenfield flow (create directory → `uv init --package` → `dev-stack init` → first commit) without encountering errors, `--force`, or `--no-verify`.
- **SC-002**: The security stage passes on 100% of freshly initialized repos that contain no actual secrets, with zero false positives from dev-stack's own managed files.
- **SC-003**: Setting `DEV_STACK_AGENT=none` results in `agent.cli = "none"` in the manifest 100% of the time, regardless of installed agents.
- **SC-004**: No `constitution-template.md` file exists at the repo root after a fresh `dev-stack init`; all baseline practices content is present in the speckit constitution template.
- **SC-005**: `dev-stack.toml` records `mode = "greenfield"` when the greenfield flow is followed with no additional user files.
- **SC-006**: A user following the README verbatim can complete every step without undocumented workarounds, confirmed by executing each instruction in a fresh environment.

## Clarifications

### Session 2026-03-10

- Q: Which file should receive baseline practices — `.specify/templates/constitution-template.md` (scaffold) or `.specify/memory/constitution.md` (active)? → A: Inject into `.specify/templates/constitution-template.md` (scaffold for new constitutions).
- Q: How should `detect-secrets` exclusions be implemented — persisted config file or CLI arguments? → A: Pass `--exclude-files` regex as CLI arguments in both `_generate_secrets_baseline()` and `_execute_security_stage()` (no config file).
- Q: Should the README document `--no-verify` as the primary first-commit step, or assume the security stage fix makes it unnecessary? → A: README documents the clean flow (no `--no-verify` needed) with a troubleshooting note for edge cases.
- Q: What should `vcs_hooks` do with baseline practices when `speckit` is not installed? → A: Skip constitution content entirely when speckit is not installed (no file created).

## Assumptions

- The `detect-secrets` tool supports `--exclude-files` as a CLI argument accepting a regex pattern to exclude matching file paths from scanning. No persistent config file is needed.
- The speckit module creates `.specify/templates/constitution-template.md` before or during the same init run as `vcs_hooks`, so the target file exists when baseline practices are injected.
- The `uv init --package` output structure (pyproject.toml with `uv_build` backend, placeholder description, `src/<pkg>/__init__.py`) is stable across current supported `uv` versions.
- The README is maintained alongside the codebase and can be updated as part of this feature branch.
